# Oriel Design System

## Overview

Oriel is a local-first macOS time tracker for people who need accurate project time without sending activity history to a hosted service. The interface exposes private app names, browser URLs, document titles, screenshots, projects, billing data, and correction workflows. It must feel trustworthy, exact, and native enough for repeated daily review.

The product personality is calm, precise, and private. UI language should be direct and literal. Visual language should feel like a high-end local macOS tool: quiet graphite surfaces, compact controls, clear hierarchy, and no decorative branding layer.

The target feel is closer to Linear, Raycast, and disciplined native macOS utilities than to a SaaS analytics dashboard. The UI should support scanning, comparison, correction, and assignment. Density is a feature. Dramatic presentation is not.

Design-system work must preserve behavior. Do not change capture logic, time calculations, stored data, AI settings, exclusions, privacy defaults, exports, Keychain behavior, Logo.dev behavior, or native messaging while normalizing UI.

The theme selector is a live feature. `graphite`, `light`, and `reference` must remain selectable unless the task explicitly changes theme behavior. `graphite` is the primary design direction, while `light` and `reference` are compatibility themes that should stay coherent and usable.

## Colors

Oriel uses semantic color roles, not page-specific color choices. Implementation tokens live in `web/css/index.css`; this file gives agents the design meaning and exact values to apply.

The default unprefixed color tokens in the YAML frontmatter represent the `graphite` theme. `light-*` and `reference-*` tokens represent the two other live selectable themes.

### Accent Rules

Blue has one job at a time. Use accent tokens for primary action, current selection, focus, links, and project identity only when the project itself uses blue. Do not use blue as generic decoration, inactive chrome, arbitrary icon color, metric emphasis, or info-card filler.

Selected state must combine at least two affordances when practical: background, border, icon, checkmark, text label, or shape. Do not communicate selected, danger, success, provider state, or AI state by color alone.

Project color swatches and chart series may use hard-coded or data-provided colors because color identity is the content. Keep those exceptions local and obvious.

## Typography

Use one product UI type family: Inter with native system fallbacks. The app is a dense desktop tool, so type should be compact, stable, and readable at normal macOS window sizes.

Do not use fluid viewport typography, `clamp()` scales, oversized marketing headings, or page-hero type inside app panels. Letter spacing should stay `0` unless an existing dense metadata treatment requires an explicit exception.

All time, duration, billing, and activity-review data should use tabular numerals through `font-variant-numeric: tabular-nums` or equivalent `font-feature-settings`.

## Layout

Oriel should feel compact without feeling cramped. Spacing should reinforce repeated desktop work: scan, compare, select, correct, assign, and verify.

The spacing scale is `2, 4, 6, 8, 12, 16, 20, 24, 32`. Use this scale before adding any new spacing value.

| Pattern | Rule |
| --- | --- |
| App chrome | 54px high and stable |
| Pane header | About 46px high when used |
| Panel padding | 20px or 24px |
| Card padding | 12px or 16px, based on density |
| Dense row padding | 8-12px vertical and 12-16px horizontal |
| Button/control height | 32px |
| Tabs | 28px active area inside a 3px padded group |
| Modal padding | 22px panel padding, 16px body rhythm |
| Footer actions | 10px gap, right aligned unless split footer is needed |

Avoid nested cards. If structure needs another level, prefer a section label, separator, table-like row group, or recessed inset surface. Cards inside cards make the app feel like a generic dashboard and reduce scan speed.

Major layout surfaces:

- App shell: full-window canvas, neutral graphite, no marketing hero or decorative background.
- Top chrome: compact native control area, date navigation, workspace tabs, and status controls.
- Timeline: dense review field with stable row height and clear selection.
- Sidebars: compact summaries and controls, not large card stacks.
- Settings: grouped controls with shared labels, helpers, callouts, and danger zones.
- Projects/statistics/AI: use shared panels and cards, but keep project/chart colors only where they encode data identity.

Modal sizing:

| Size class | Width | Use |
| --- | --- | --- |
| `modal-size--sm` | 360px | Small utility forms |
| `modal-size--md` | 420px | Standard forms |
| `modal-size--lg` | 650px | Time entry and wider details |
| `modal-size--xl` | 750px | Settings and complex dialogs |
| `modal-size--split` | 800px | Split inspection/editing layouts |
| `modal-size--confirm` | 380px | Confirmations |

Tall modals should be top-aligned below the app chrome. Compact confirmations may be centered.

## Elevation & Depth

Depth is mostly tonal, not shadow-heavy. Use surface progression first: canvas, panel, raised, recessed, hover. Use borders and separators for structure. Use shadows only for floating UI such as modals, popovers, dropdowns, and tooltips.

Implementation elevation tokens:

```css
--shadow-float: 0 16px 40px oklch(0.08 0.004 260 / 0.44), 0 1px 0 oklch(0.9 0.005 260 / 0.03) inset;
--shadow-popover: var(--shadow-float);
```

| `--radius-md` | 8px | Buttons, fields, selects, icon buttons |
| `--radius-lg` | 10px | Cards, panels, row groups |
| `--radius-xl` | 12px | Modals and popovers |
| `--radius-pill` | 999px | Pills, dots, badges, swatches |

Do not introduce oversized rounded cards. Do not mix sharp and soft corners inside the same component family.

## macOS Window Chrome

Oriel is a macOS desktop app with a web UI inside the native shell. Chrome should defer to macOS conventions and stay compact.

Rules:

- App chrome height is 54px.
- Native shell mode reserves left padding for traffic lights.
- Traffic light positioning belongs to native window code, not CSS decoration.
- Keep toolbar controls visually quiet; the current task should be obvious without a large app title.
- Status bar and tracking status controls should be compact, neutral, and explain state with tooltip text.
- Do not add custom draggable hero regions, large brand blocks, or decorative titlebar artwork.

When a UI task touches native window chrome, inspect the SwiftUI/AppKit code and the web chrome together. The web UI must not fight the native titlebar or traffic-light hit area.

## Scrollbar

Scrollbars are part of the design system because Oriel contains long timelines, settings lists, provider lists, and modal bodies.

Rules:

- Scrollbar size is 8px.
- Track uses the recessed surface.
- Thumb uses strong border mixed with hover surface.
- Thumb hover increases contrast without becoming an accent color.
- Corners use the recessed surface.
- Use `scrollbar-gutter: stable` where layout shift would hurt scanning.

Theme values are tokenized as `scrollbar-track`, `scrollbar-thumb`, `scrollbar-thumb-hover`, and `scrollbar-corner`, plus `light-*` and `reference-*` variants in YAML.

## Motion

Motion should be functional and brief. The default easing is:

```css
--ease-out: cubic-bezier(0.165, 0.84, 0.44, 1);
```

Default transitions should be about 160ms for background, border, color, and transform feedback. Panel entrance may use about 170ms. Loading spinner motion may use `ai-spin` at 800ms linear.

Motion rules:

- Use motion to confirm a state change, opening surface, hover, active press, or loading state.
- Do not add ambient motion, decorative animations, bouncing, elastic transitions, or large transform choreography.
- Respect reduced-motion settings for non-essential animation.
- Keep timeline interactions stable; motion must not make time ranges harder to read.

## Components

Components should be CSS class contracts and small existing JS template/helper cleanups. Do not add a new UI framework for component consistency.

### Buttons

Use `button-primary`, `button-secondary`, `button-danger`, and `icon-button`.

Rules:

- Height: 32px for text buttons.
- Icon button hit area: 30px by 30px.
- Radius: `--radius-md`.
- Font: 12px / 16px, weight 500.
- Text button horizontal padding: 13px.
- Gap between icon and label: 7px.
- Every button needs default, hover, focus-visible, active, disabled, and loading affordances where applicable.
- Primary buttons are for the main action in a local context. Avoid multiple primary buttons in one small region.
- Destructive actions use danger tokens and clear labels.
- Loading actions should expose `aria-busy` or an equivalent state when possible.

### Icon Buttons

Icon-only controls need an accessible label or title. Use Phosphor icons already bundled with the app. Do not manually draw icons when an existing icon exists.

Icon active state uses accent text, accent border, and accent wash. Hover uses neutral hover surface unless the active state is selected.

### Fields And Selects

Use `field`, `custom-select`, and app-rendered select menus.

Rules:

- Height: 32px.
- Radius: `--radius-md`.
- Background: `--control-surface`.
- Hover background: `--control-surface-hover`.
- Border: `--border`, stronger on hover.
- Placeholder: `--text-tertiary`.
- Disabled opacity: about 0.62 for fields and selects.
- Focus ring: 2px solid `--focus-ring`, offset by `--focus-ring-offset`.

Avoid native select popups when the app-rendered menu pattern is already established.

### Tabs And Segmented Controls

Use `app-tab-group`, `app-tab`, `range-group`, and `range-pill`.

Rules:

- Group border: `--border`.
- Group radius: `--radius-lg`.
- Group padding: 3px.
- Tab active area: 28px minimum height.
- Tab radius: 7px.
- Active tab uses raised surface and selected shadow.
- Hover and focus use panel or hover surface, not accent fill.

### Modals

Use one modal shell vocabulary for all dialogs.

Rules:

- Overlay uses `--overlay` and `--z-modal`.
- Tall modals top-align below app chrome with `calc(54px + 8px)` top padding.
- Compact confirmation modals may use centered alignment.
- Panel uses `--modal-surface`, `--separator-strong`, `--radius-xl`, and `--shadow-popover`.
- Panel padding is 22px.
- Header has a 12px bottom gap and separator.
- Title uses 16px / 22px, weight 700.
- Body is a vertical flex column with 16px gap.
- Footer has 16px top padding, 10px action gap, and separator.
- Close buttons must use the shared icon-button treatment.

Do not let each modal invent its own header, footer, close-button style, width, callout treatment, or danger-action style.

### Panels And Cards

Use `workspace-panel` for major app regions and `surface-panel` for grouped controls, sidebar modules, and compact cards.

Rules:

- Major regions should feel like panes inside the app, not marketing cards.
- Cards use raised or panel surface, subtle border, 10px radius, and 12px or 16px padding.
- Hoverable cards should change surface or border subtly.
- Avoid nested cards. Use separators, section labels, or recessed rows instead.
- Stat cards, project cards, report panels, AI cards, and sidebar cards should converge on the same surface model.

### Badges, Pills, Dots, And Metadata

Use shared pill styles for durations, statuses, activity mix, and metadata.

Rules:

- Radius: pill.
- Use metadata typography.
- Pair status color with text, icon, position, or shape.
- Project color dots are allowed because project color identity is content.
- Duration pills should remain compact and tabular.

### Callouts And Danger Zones

Info callouts should be neutral or info-token based, not arbitrary blue boxes. Danger zones use danger tokens, clear labels, and explicit copy about irreversible behavior.

Any callout involving AI providers, Logo.dev, screenshots, Keychain, capture permissions, storage, exports, or browser data must make the privacy/security impact visible in the UI and PR description.

### Empty States

Empty states should be quiet, compact, and actionable when an action exists. Do not use decorative illustrations, oversized icons, or large marketing copy. The best empty state tells the user what is currently true and offers the next useful action.

### Timeline Entries And Activity Rows

Timeline density is important and should be preserved.

Rules:

- Default row height is 40px.
- Rows must support fast scanning of app, title, domain, time, duration, assignment, and status.
- Hover, selection, drag, and multi-select states must remain visually distinct.
- Assignment actions should not obscure the time data needed to make the assignment.
- Activity popovers must use popover elevation, compact metadata typography, and stable z-index layers.

## Accessibility

Target WCAG 2.2 AA. This is especially important because Oriel contains dense data and privacy-sensitive state.

Rules:

- Normal text should meet 4.5:1 contrast against its background.
- Large or bold text should meet at least 3:1 contrast.
- Focus-visible must be clear on every button, field, select, tab, menu option, and close control.
- Do not rely on color alone for selected, active, danger, success, warning, provider state, AI state, capture state, or billable state.
- Icon-only controls require accessible names.
- Loading states require visible text, disabled/progress affordance, or `aria-busy` when practical.
- Reduced-motion preferences must be respected for non-essential motion.
- Text must not overflow controls, overlap neighboring content, or resize fixed-format UI unexpectedly.

Privacy-sensitive UI states need plain labels. For example, AI provider status, Logo.dev fetching, screenshot behavior, browser-extension capture, exclusions, and Keychain-backed secrets must be understandable without inspecting color.

## Do's and Don'ts

- Do read this file before UI work.
- Do treat `web/css/index.css` as the implementation token source.
- Do preserve `graphite`, `light`, and `reference` while the theme selector remains live.
- Do use unprefixed YAML color tokens as the default graphite theme.
- Do use `light-*` and `reference-*` tokens when documenting or implementing theme-specific overrides.
- Do keep graphite as the primary design direction unless the task explicitly changes theme strategy.
- Do use semantic tokens for surface, text, border, accent, status, overlay, focus, and selection.
- Do keep desktop density and scanning speed high.
- Do use one component vocabulary for buttons, fields, selects, tabs, panels, cards, badges, callouts, and modals.
- Do keep blue accent usage disciplined and explainable.
- Do use project/chart colors only when color identity is the data.
- Do include privacy/security notes in PRs that touch capture, exclusions, storage, screenshots, exports, Keychain, Native Messaging, AI, Logo.dev, browser URLs, or logging.
- Don't add new colors, spacing, typography, radii, shadows, z-index values, motion curves, or component variants without updating this file intentionally.
- Don't redesign `light` or `reference` as separate product directions unless the task explicitly asks for theme redesign.
- Don't use oversized headings, hero layouts, decorative gradients, glow, glass effects, or marketing-card rhythm.
- Don't add nested cards where a label, divider, row group, or inset surface would be clearer.
- Don't use blue for arbitrary icons, generic card decoration, inactive borders, or metric decoration.
- Don't communicate state by color alone.
- Don't commit screenshots, local activity data, SQLite files, logs, archives, credentials, signing material, private URLs, or user-identifying fixtures.
