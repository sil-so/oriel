# Product

## Register

product

## Users

Oriel is for Mac users who need accurate project time without sending activity
history to a hosted service. Typical users are freelancers, consultants,
builders, and independent operators who review captured app/browser activity,
turn it into project entries, and use the results for billing, reporting, and
personal accountability.

Users work in a dense desktop context with private app names, document titles,
URLs, screenshots, projects, and billing data visible on screen. The interface
must support repeated review, correction, and assignment work without feeling
decorative or fragile.

## Product Purpose

Oriel records local foreground app and browser activity, presents it as a
reviewable Activity Stream Timeline, and helps users convert raw activity into
project, task, billable, and reporting entries. Success means users can trust
the capture history, quickly understand where time went, and confidently assign
or correct time without exposing private data by default.

## Brand Personality

Calm, precise, private.

The product voice should be direct and understated. UI copy should describe
what Oriel literally does, avoid marketing language, and make privacy-sensitive
actions explicit.

## Anti-references

- A generic SaaS analytics dashboard with oversized metrics, decorative cards,
  or marketing-page hierarchy.
- Decorative glass, glow, gradients, or blue accents used as visual filler.
- A theme showcase where multiple unfinished themes multiply inconsistencies.
- Modal and settings screens that each invent their own spacing, title, footer,
  callout, and control treatment.
- Interfaces that make local activity data feel casual, public, or cloud-first.

## Design Principles

- Trust local data first: privacy, capture state, exclusions, and optional
  network behavior must stay explicit.
- Keep the tool dense and native-feeling: prioritize scanning, comparison, and
  repeated desktop workflows over dramatic presentation.
- Use one component vocabulary: the same action, field, card, panel, modal, and
  selected state should look the same everywhere.
- Give blue one job at a time: reserve it for primary action, current
  selection, focus, links, and project identity where needed.
- Preserve behavior while normalizing UI: design-system work should not alter
  time calculations, capture logic, stored formats, AI settings, or privacy
  defaults.

## Accessibility & Inclusion

Target WCAG 2.2 AA for contrast, visible focus, keyboard access, and non-color
state communication. Prefer fixed product UI type scales, clear focus rings,
and reduced-motion-safe state transitions. Do not rely on color alone for
status, selection, danger, success, or AI/provider configuration state.
